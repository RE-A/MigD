## 1. 소개

MigD는 QA DB(소스)에서 개발/테스트용 DB(대상)로 **원하는 테이블만 골라 데이터를 복사**하는 도구다.

**주요 특징**:
- 이관할 테이블과 WHERE 조건을 자유롭게 지정
- 자주 쓰는 테이블 조합을 프리셋으로 저장해 재사용
- 소스 DB는 서버에 미리 고정 설정 — 사용자가 직접 입력하지 않음
- 대상 DB는 이관 화면에서 매번 직접 입력
- AWS RDS 등 클라우드 주소는 대상으로 사용 불가 (실수 방지)

## 2. 접속 및 화면 구성

### 2.1 접속

브라우저에서 `http://서버주소:8080` 으로 접속한다.

### 2.2 대시보드 (`/`)

접속하면 대시보드가 표시된다. 상단 내비게이션 바에서 모든 기능으로 이동할 수 있다.

```
[MigD]  이관 실행  스키마 이관  프리셋 관리
```

대시보드에는 최근 저장된 프리셋 목록이 표시된다.

---

## 3. 프리셋 관리

프리셋이란 **이관할 테이블 목록을 저장해 둔 것**이다. 동일한 테이블 조합을 반복해서 이관할 때 매번 입력하지 않아도 된다.

### 3.1 프리셋 목록 (`/presets`)

저장된 프리셋 목록을 확인하고 다음 작업을 할 수 있다:

| 버튼 | 기능 |
|------|------|
| 이관 실행 | 해당 프리셋의 테이블 목록을 불러와 이관 화면으로 이동 |
| 스키마 이관 | 스키마 이관 화면으로 이동 |
| 수정 | 프리셋 편집 |
| 삭제 | 프리셋 삭제 (복구 불가) |

### 3.2 프리셋 생성

**내비게이션 바 → 프리셋 관리 → 새 프리셋** 또는 목록 화면의 **+ 새 프리셋** 버튼을 클릭한다.

**기본 정보 입력**

| 필드 | 설명 | 필수 |
|------|------|------|
| 프리셋 이름 | 식별용 이름. 중복 불가 | ✓ |
| 설명 | 어떤 용도인지 메모 | - |

**테이블 추가**

테이블을 추가하는 방법은 두 가지다:

**방법 1 — 직접 입력**: 스키마, 테이블명, WHERE 조건(선택), 순서를 직접 입력하고 **+ 행 추가** 버튼을 클릭한다.

**방법 2 — 검색 추가**: 상단 검색창에 테이블명 일부를 입력하면 목록이 필터링된다. 원하는 테이블의 체크박스를 선택 후 **선택 항목 추가** 버튼을 클릭한다.

> **WHERE 조건 예시**
> - `status = 'ACTIVE'`
> - `created_at >= '2024-01-01'`
> - `dept_id = 1 AND is_active = TRUE`
> - (비워두면) 테이블 전체 데이터 이관

테이블 목록에서 **↑↓** 버튼으로 실행 순서를 조정하고, **×** 버튼으로 행을 삭제할 수 있다.

**저장**: 오른쪽 하단의 **저장** 버튼을 클릭한다.

### 3.3 프리셋 수정

목록에서 **수정** 버튼을 클릭해 기존 프리셋을 편집한다. 저장 시 테이블 목록 전체가 교체된다.

## 4. 데이터 이관 실행

### 4.1 이관 화면 열기

두 가지 방법으로 이관 화면을 열 수 있다:

1. **내비게이션 바 → 이관 실행** — 빈 상태로 시작
2. **프리셋 목록 → 이관 실행 버튼** — 해당 프리셋의 테이블 목록이 자동으로 채워진 상태로 시작

### 4.2 소스 DB 확인

화면 상단에 소스 DB(QA DB) 접속 정보가 표시된다. 이 정보는 서버에 고정 설정된 것으로, **사용자가 변경할 수 없다**. 잘못된 정보라면 서버 관리자에게 문의한다.

### 4.3 대상 DB 입력

이관 대상이 될 DB 접속 정보를 입력한다.

| 필드 | 예시 | 설명 |
|------|------|------|
| 호스트 | `dev-db.internal` | DB 서버 주소 |
| 포트 | `5432` | 기본값 5432 |
| 데이터베이스 | `devdb` | DB 이름 |
| 사용자 | `devuser` | 접속 계정 |
> **AWS 주소 사용 불가**: `*.amazonaws.com`, `*.rds.*`, `*.elasticache.*` 등 AWS 도메인은 입력해도 차단된다.

### 4.4 테이블 목록 구성

**프리셋 불러오기**: 화면 우측의 프리셋 드롭다운에서 프리셋을 선택하고 **불러오기** 버튼을 클릭하면 테이블 목록이 자동으로 채워진다. 불러온 후에도 테이블을 추가하거나 삭제할 수 있다.

**직접 구성**: 테이블 목록 하단의 **+ 행 추가** 버튼으로 테이블을 하나씩 추가한다.

| 컬럼 | 설명 |
|------|------|
| 스키마 | 테이블의 PostgreSQL 스키마명 (예: `public`, `hr`) |
| 테이블명 | 이관할 테이블명 |
| WHERE 조건 | 이관 범위를 제한할 SQL 조건. 비워두면 전체 이관 |

**현재 목록 프리셋으로 저장**: 오른쪽 이름 입력란에 프리셋 이름을 입력하고 **저장** 버튼을 클릭하면 현재 테이블 목록이 새 프리셋으로 저장된다.

### 4.5 이관 실행

**이관 실행** 버튼을 클릭하면 이관이 시작된다.

- 테이블은 목록에 표시된 순서대로 처리된다
- 각 테이블은 독립적인 트랜잭션으로 처리된다
- 한 테이블이 실패하면 이후 테이블은 실행되지 않고 건너뜀 처리된다

## 5. 스키마 이관

**내비게이션 바 → 스키마 이관** 또는 프리셋 목록의 **스키마 이관** 버튼으로 접근한다.

### 5.1 용도

대상 DB에 스키마(테이블, 시퀀스, 인덱스, 함수, 프로시저 등)를 복사한다. 데이터는 이관하지 않는다. 대상 DB가 새로 구성되었거나 구조가 맞지 않을 때 사용한다.

### 5.2 사용 방법

1. 대상 DB 접속 정보 입력 (데이터 이관과 동일)
2. **스키마 이름** 입력 (예: `public`, `hr`) — 해당 스키마의 전체 오브젝트가 대상 DB에 적용됨
3. **스키마 이관 실행** 버튼 클릭

### 5.3 스키마 이관과 데이터 이관의 차이

| | 스키마 이관 | 데이터 이관 |
|---|------------|------------|
| 대상 | 스키마 전체 (테이블 구조, 함수, 인덱스 등) | 지정한 테이블의 데이터 |
| 데이터 | 이관하지 않음 | 이관함 |
| 단위 | 스키마 전체 | 테이블 단위 |
| WHERE 조건 | 없음 | 가능 |

### 5.4 권장 워크플로

새 DB에 처음 이관할 때의 순서:

```
1. 스키마 이관 → 빈 DB에 구조 생성
2. 데이터 이관 → 원하는 테이블의 데이터 복사
```

## 6. 이관 결과 확인

이관 완료 후 결과 화면이 표시된다.

### 6.1 결과 화면 구성

**요약 정보**:
- 대상 DB 호스트/DB명
- 성공 테이블 수 / 실패 테이블 수
- 총 이관 행 수

**테이블별 결과 목록**:

| 상태 | 의미 |
|------|------|
| ✓ 성공 | 정상 이관 완료. N건 표시 |
| ✗ 실패 | 이관 실패. 오류 메시지 표시 |
| - 건너뜀 | 이전 테이블 실패로 미실행 |

**오류 메시지 예시**:

```
PK 중복 오류 [public.users]: Target DB에 이미 데이터가 존재합니다.
직접 삭제 후 재시도하세요.
```

→ 대상 테이블의 기존 데이터를 먼저 삭제한 뒤 재실행해야 한다.

### 6.2 결과 후 행동

| 버튼 | 이동 위치 |
|------|---------|
| 다시 실행 | 이관 실행 화면 (빈 상태) |
| 대시보드 | 홈 화면 |

## 7. 주의사항 및 제약

### 7.1 PK 중복 처리

대상 테이블에 **이미 동일한 PK를 가진 데이터가 있으면 이관이 실패**한다. Upsert(덮어쓰기)를 지원하지 않는다.

해결 방법: 대상 테이블의 기존 데이터를 직접 삭제한 뒤 재실행한다.

```sql
-- 대상 DB에서 직접 실행
DELETE FROM "public"."your_table";
-- 또는 특정 범위만
DELETE FROM "public"."your_table" WHERE id IN (...);
```

### 7.2 외래 키(FK) 없는 환경 전제

MigD는 FK 제약이 없는 환경을 전제한다. 테이블 실행 순서는 사용자가 직접 지정한다. FK가 있는 DB에서 이관하면 참조 무결성 오류가 발생할 수 있다.

### 7.3 이관 중단 시 동작

한 테이블이 실패하면 이후 테이블은 모두 건너뛴다. 이미 성공한 테이블의 데이터는 커밋된 상태로 유지된다 (자동 롤백되지 않음).

### 7.4 AWS 클라우드 주소 사용 불가

다음 패턴의 호스트는 대상 DB로 사용할 수 없다:
- `*.amazonaws.com`
- `*.rds.*`
- `*.elasticache.*`
- `*.aws.*`

실수로 운영 DB에 이관하는 것을 방지하기 위한 제한이다.

### 7.5 스키마 이관의 DDL 적용 제한

스키마 이관(`applyFullSchemaDump`)은 pg_dump 출력에서 `--`, `SET`, `BEGIN`, `COMMIT` 라인을 제거 후 적용한다. 복잡한 함수/프로시저 본문에 세미콜론이 포함된 경우 DDL이 올바르게 적용되지 않을 수 있다. 이 경우 pg_dump를 직접 실행해 psql로 적용하는 것을 권장한다.

### 7.6 대용량 이관

COPY BINARY 스트리밍 방식을 사용하므로 대용량 테이블도 64KB 단위로 스트리밍되어 메모리 문제 없이 이관된다. 단, 네트워크 속도에 따라 시간이 걸릴 수 있다.

## 8. 자주 묻는 질문

**Q. 소스 DB를 변경하고 싶다**

서버의 `application.yml`에서 `migd.source-db.*` 항목을 수정하고 서버를 재시작해야 한다. 화면에서는 변경할 수 없다.

**Q. 이관 중에 화면을 닫으면 어떻게 되나?**

이관은 서버에서 실행되므로 화면을 닫아도 계속 진행된다. 결과는 브라우저를 닫기 전에 확인해야 한다 (현재 진행 상황을 다시 조회하는 기능 없음).

**Q. WHERE 조건에 서브쿼리를 사용할 수 있나?**

가능하다. 단, 소스 DB의 테이블을 참조하는 서브쿼리여야 한다.

```
id IN (SELECT user_id FROM orders WHERE status = 'ACTIVE')
```

**Q. 동일 테이블을 WHERE 조건만 다르게 여러 번 이관할 수 있나?**

한 번의 실행에서는 동일 테이블을 두 번 지정할 수 없다. 하지만 첫 번째 이관 후 대상 테이블에 데이터가 있는 상태에서 다른 WHERE 조건으로 다시 이관하면 PK 중복 오류가 발생할 수 있다.

**Q. 프리셋은 어디에 저장되나?**

서버의 `./data/preset_db` 경로에 H2 파일 DB로 저장된다. 서버를 재시작해도 유지된다.

**Q. 이관 속도가 너무 느리다**

소스/대상 DB 서버와의 네트워크 대역폭이 주요 병목이다. 또한 테이블당 1개의 Connection을 사용하므로 다수의 소형 테이블보다 대형 테이블이 더 효율적이다.

**Q. pg_dump를 찾을 수 없다는 오류가 발생한다**

스키마 이관 기능은 로컬에 설치된 `pg_dump` 바이너리를 사용한다. `application.yml`의 `migd.pg-dump-path`가 올바른 경로를 가리키는지 확인한다.

```yaml
migd:
  pg-dump-path: "C:/Program Files/PostgreSQL/16/bin/pg_dump.exe"
```
